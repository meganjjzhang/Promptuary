# MultiAIEdit — 竞争分析与场景替代评估

> 日期：2026-06-10 | 评估维度：迁移动力、场景闭环、替代方案

---

## 一、能否吸引已有三插件的用户？

### 结论：迁移存量用户不是增长路径

三插件合计用户量小（~380），且需求分化严重。**"AI 批量执行"是新需求，不是已有需求的升级**。

### 逐个分析

#### Sidebar Highlights 用户（~137 人）

| 维度 | 分析 |
|------|------|
| 核心诉求 | 高亮分类管理 + 跨文件搜索 + 集合组织 |
| 迁移动力 | 如果他们**同时**想 AI 批量修改 → 有强动力 |
| 迁移阻力 | 不需要 AI 执行时，当前方案已满足 → 无动力 |
| **关键矛盾** | Sidebar HL 改原文，有人认为是 bug，有人认为是 feature。认为 feature 的人不会走 |
| 迁移率预估 | **20-30%**（仅"分类管理 + AI 执行"交集用户） |

#### HiNote 用户（~241 人）

| 维度 | 分析 |
|------|------|
| 核心诉求 | AI 注释 + 导出为知识卡片/笔记 + 间隔重复闪卡 |
| 迁移动力 | 如果 AI 注释不够，想要 AI 执行修改 → 有动力 |
| 迁移阻力 | HiNote 的导出图片/闪卡是我们的弱项；他们的用户更偏"学习"而非"修改" |
| **关键矛盾** | HiNote 用户画像偏"知识管理"，我们偏"文档修改"，需求错位 |
| 迁移率预估 | **15-25%**（仅"AI 执行"强需求用户） |

#### Axl Light 用户（~1 人）

不具备迁移分析价值。但其 sidecar 架构思路值得借鉴。

### 重新定位增长路径

**不能靠迁移存量用户，要靠吸引新用户**：

- 目标人群：已经在用 ChatGPT/Claude 改文档，但觉得体验割裂的人
- 触达方式：Obsidian 社区论坛、Reddit r/ObsidianMD、小红书"Obsidian AI"
- 核心钩子：**"在 Obsidian 里直接让 AI 批量修改文档，还有 Diff 预览"**

## 二、场景闭环性分析

### 链路闭环检查

| 环节 | 闭环性 | 外部依赖 | 风险 |
|------|--------|---------|------|
| 批注录入 | ✅ 自闭环 | 无 | — |
| 侧边栏汇总 | ✅ 自闭环 | 无 | — |
| Prompt 生成 | ✅ 自闭环 | 无 | — |
| **AI 执行** | **⚠️ 不闭环** | **API Key + 联网 + Token 费用** | **核心链路依赖外部** |
| Diff 确认 | ✅ 自闭环 | 无 | — |

### AI 执行环节的三个依赖

1. **API Key 门槛**：用户需自备 OpenAI / Anthropic / DeepSeek 的 API Key
   - 大量 Obsidian 用户没有 API Key，也不知道怎么获取
   - 这和"Obsidian 里自然使用"的体验期望冲突

2. **网络依赖**：除 Ollama 本地模式外均需联网
   - Obsidian 用户中有相当比例的"离线偏好"用户

3. **Token 费用**：长文档每次执行消耗 $0.01-0.10
   - 频繁批阅场景下费用可感知
   - 用户心理："一个 Obsidian 插件还要花钱？"

### 降低闭环断裂风险的方案

| 方案 | 描述 | 闭环性 | 门槛 |
|------|------|--------|------|
| **A. Prompt 导出模式** | 生成 Prompt 复制到 ChatGPT 网页版 | 半闭环 | 零 |
| B. Ollama 本地模型 | 零费用、离线可用 | 闭环 | 需 GPU |
| C. DeepSeek 低价模型 | 极低费用 | 闭环 | 需 API Key |
| D. 内置免费额度 | 插件提供有限免费 API 调用 | 闭环 | 运营成本高 |

**关键洞察：方案 A（Prompt 导出模式）最务实**——即使不用 API，"批注 → 生成结构化 Prompt → 复制到 ChatGPT" 本身就有独立价值，且零门槛。应该作为 MVP 的默认模式，API 直调作为高级模式。

## 三、用户场景 vs 替代方案

### 场景 1：批阅产品文档（修改点密集）

| | MultiAIEdit | 替代方案：复制到 ChatGPT |
|--|------------|----------------------|
| 操作 | 选中批注 → 一键执行 | 复制文档 → 描述修改 → 复制回来 |
| 门槛 | 需安装+配置 API | 零门槛 |
| 差异对比 | ✅ Diff 预览 | ❌ 手动对比 |
| 批注管理 | ✅ 结构化、可回溯 | ❌ 散落在聊天里 |
| 遗漏风险 | ✅ 侧边栏不遗漏 | ❌ 多轮对话易遗漏 |
| **胜算** | **能赢** | 但需用户先感知"盲改不安全"痛点 |

### 场景 2：审阅周报/报告

| | MultiAIEdit | 替代方案：飞书/Google Docs 协作批注 |
|--|------------|----------------------------------|
| 操作 | 选中批注 → AI 执行 | 在线批注 → 建议模式 |
| 平台 | ✅ Obsidian 本地文件 | ❌ 仅限在线文档 |
| AI 能力 | ✅ AI 批量执行修改 | ❌ 无 AI 执行 |
| 协作 | ❌ 单人工具 | ✅ 多人协作天然支持 |
| **胜算** | **能赢（本地文件场景）** | 在线文档场景打不过 |

### 场景 3：阅读技术文档标记问题

| | MultiAIEdit | 替代方案：Obsidian %%注释%% 或 Sidebar Highlights |
|--|------------|------------------------------------------------|
| 操作 | 选中 → 批注面板 | 选中 → 内联注释 |
| 门槛 | 需安装插件 | 零门槛（%% 原生支持） |
| 汇总 | ✅ 侧边栏 | ❌ 无汇总（%% 模式） / ✅ 有汇总（Sidebar HL） |
| 执行 | ✅ AI 执行 | ❌ 无 |
| **胜算** | **难赢** | 如果只标记不执行，轻量方案更受欢迎 |

**反思**：如果用户只是标记问题、不立即修改，这个插件就是杀鸡用牛刀。需要在产品定位上明确——我们解决的是"批注后要执行修改"的场景，不是"标记问题"的场景。

### 场景 4：多文件统一修改

| | MultiAIEdit | 替代方案：Obsidian 搜索替换 / VS Code |
|--|------------|-------------------------------------|
| 操作 | 批注 → AI 执行 | Ctrl+H 批量替换 |
| 精确匹配 | ❌ AI 可能改偏 | ✅ 精确替换 |
| 语义修改 | ✅ 自然语言描述 | ❌ 只能精确匹配 |
| **胜算** | **能赢（语义修改场景）** | 精确替换场景打不过 |

**关键**：语义级修改是我们的独有能力。"把所有'用户'替换为'客户'"用搜索替换更好；"把所有过于正式的表述改为更口语化的风格"只有 AI 能做。

### 场景 5：学术论文批注

| | MultiAIEdit | 替代方案：Zotero + PDF 阅读器 |
|--|------------|------------------------------|
| 操作 | 在 MD 上批注 → AI 修改 | 在 PDF 上批注 → 手动修改 |
| PDF 支持 | ❌ 仅 Markdown | ✅ 原生 PDF 批注 |
| 学术生态 | ❌ 弱 | ✅ Zotero 生态成熟 |
| AI 能力 | ✅ AI 执行 | ❌ 无 |
| **胜算** | **小赢** | 仅在 MD 化之后的修改环节有优势 |

### 场景 6：最强竞争者——ChatGPT/Claude 网页版

这是我们必须正视的**最大竞争者**。

用户最自然的替代路径：
```
复制文档 → 粘贴到 ChatGPT → 描述修改 → 复制回来
```

**ChatGPT 的优势**：
- 零安装、零配置、免费（免费版够用）
- 用户心智已建立——"改文档？丢给 ChatGPT"
- 多轮对话灵活

**ChatGPT 的弱点**（我们能赢的点）：
1. **盲改无 Diff**：ChatGPT 给你改完的整篇文档，你很难精确对比改了哪里
2. **批注散落**：多轮对话中修改指令散落，容易遗漏
3. **上下文丢失**：长文档超过上下文窗口就截断
4. **不可回溯**：修改历史在聊天记录里，无法结构化回看

## 四、战略建议

### 1. 重新定义 MVP 闭环

**不要让 AI API 成为使用的硬门槛。** 建议调整 MVP 设计：

```
MVP 默认模式（零门槛）：
  批注录入 → 侧边栏汇总 → 一键生成 Prompt → 复制到剪贴板
  用户粘贴到 ChatGPT → 得到修改结果 → 粘贴回 Obsidian

MVP 高级模式（需 API Key）：
  批注录入 → 侧边栏汇总 → AI 直调 → Diff 预览 → 确认写入
```

**"Prompt 导出"作为零门槛入口，"AI 直调"作为付费升级。**

### 2. 聚焦杀手场景

不是所有场景都值得打。按"不可替代性"排序：

| 优先级 | 场景 | 不可替代性 | 原因 |
|--------|------|-----------|------|
| **P0** | 批阅产品文档（修改密集） | 高 | Diff + 批注管理是 ChatGPT 做不到的 |
| **P0** | 语义级多文件修改 | 高 | 搜索替换做不到语义修改 |
| P1 | 审阅本地文档 | 中 | 飞书/Docs 不覆盖 Obsidian 本地场景 |
| P2 | 阅读标记问题 | 低 | %% 注释和 Sidebar HL 足够用 |
| P3 | 学术 PDF 批注 | 低 | Zotero 生态壁垒高 |

### 3. 产品定位修正

**原定位**：Obsidian 中第一个实现「阅读批注 → AI 批量修改」完整闭环的插件

**修正为**：**在 Obsidian 里用 AI 批量修改文档，每条修改看得见、管得住**

差异：
- 从"批注插件"重新定位为"AI 修改安全网"
- 核心卖点不是"能批注"（别人也能），而是"改了什么一目了然"
- 唤醒痛点：ChatGPT 给你改了一篇文档，你知道它到底改了哪里吗？

### 4. 核心竞争力矩阵

```
                ChatGPT    Sidebar HL    HiNote    MultiAIEdit
批注不污染原文    ❌          ❌           ❌         ✅
结构化批注管理    ❌          ✅           部分       ✅
AI 执行修改       ✅          ❌           ❌         ✅
Diff 预览确认     ❌          ❌           ❌         ✅
零门槛使用        ✅          ✅           部分       需优化 ⚠️
```

**我们最强的两个词：Diff 预览 + 批注管理。** 这两个能力是 ChatGPT 和其他插件都没有的。

### 5. 风险与对冲

| 风险 | 严重性 | 对冲策略 |
|------|--------|---------|
| ChatGPT 门槛太低，用户不愿安装插件 | 高 | Prompt 导出模式零门槛入口，先用起来再升级 |
| AI API 门槛筛掉大量用户 | 高 | 默认 Prompt 导出 + DeepSeek 低价引导 |
| Obsidian 社区小，天花板低 | 中 | 可扩展为 VS Code 插件（语义修改场景更大） |
| 用户感知不到"盲改不安全"痛点 | 中 | 内容营销：展示"ChatGPT 改了什么你看不出来"的案例 |
